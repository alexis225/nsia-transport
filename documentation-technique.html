Principe multi-tenant

Chaque filiale est un tenant identifié par un UUID (tenant_id). L'isolation applicative repose sur deux mécanismes complémentaires :

Isolation au niveau requête

Middleware EnsureTenantIsolation (alias tenant.isolation) sur tout le groupe de routes authentifiées, combiné à un filtrage manuel par tenant_id dans chaque contrôleur (pas de global scope Eloquent automatique — vérifié dans tous les contrôleurs admin).

Modules activables par filiale

Tenant::MODULES (16 clés) + middleware module: (EnsureModuleEnabled) permettent de masquer des pans entiers de l'application par filiale (ex : une filiale sans courtiers désactive le module brokers). Voir section Modules activables.

Row Level Security PostgreSQL — état actuel : la migration 2026_06_01_000001_enable_row_level_security.php et le middleware SetTenantContext (alias tenant.context, SET app.current_tenant_id) existent dans le code, mais ce middleware n'est actuellement rattaché à aucun groupe de routes dans routes/web.php. L'isolation multi-tenant réellement active aujourd'hui est donc uniquement applicative (middleware + filtrage tenant_id par contrôleur), pas renforcée par des politiques RLS actives en base. À vérifier/rattacher avant de considérer la RLS comme une couche de défense opérationnelle.

Architecture de SécuritéDéfense en profondeur

1. Couche réseau

  • Middleware CheckIpBlacklist (global web, en tête de stack)
  • Cache 5 min par IP (performance)
  • Support CIDR PostgreSQL ::inet <<=
  • Verrouillage compte : échecs de connexion → blocage temporaire

2. Couche authentification

  • Fortify : Bcrypt + rate limiting
  • MFA TOTP (Fortify + google2fa)
  • SSO Microsoft (Azure AD single-tenant) + Google
  • Sessions PHP + CSRF token

3. Couche autorisation

  • Spatie Permission — 6 rôles, ~90 permissions
  • Middleware permission: / role: sur routes
  • Délégation temporaire via UserRoleGrant
  • EnsureStaffAccess : isole espace admin / espace partenaire

4. Couche données

  • Isolation applicative tenant_id (RLS non rattachée — voir §1)
  • encrypted cast sur champs PII (adresse, téléphone, n° immatriculation)
  • UUID v4 pour toutes les PKs (HasUuids)
  • Soft delete (audit trail) sur les entités métier

Flux d'autorisation par rôle

6 rôles Spatie sont seedés par RolesAndPermissionsSeeder : super_admin, admin_filiale, souscripteur, courtier_local, partenaire_etranger, client. Les deux derniers rôles courtier/partenaire sont redirigés vers l'espace /partner dédié par EnsureStaffAccess.

// app/Http/Middleware/EnsureStaffAccess.php
if (
    $user->hasAnyRole(['courtier_local', 'partenaire_etranger'])
    && ! $user->hasAnyRole(['super_admin', 'admin_filiale', 'souscripteur'])
) {
    return redirect()->route('partner.dashboard');
}

Module Escalade NN300 — RenforcementArchitecture détaillée du workflow d'approbation

Le mécanisme d'escalade repose sur des règles (ApprovalWorkflowConfig, table approval_workflows) évaluées à la soumission d'un certificat. Une filiale peut définir plusieurs règles ; findForCertificate() retient la première dont la condition est vraie (sémantique OR entre les lignes de configuration).

Types de déclencheurs (trigger_condition)

Clé JSONConditionPortée
insured_value_pct_of_contractValeur du certificat > X % du plein du contrat (défaut 15 %, surchargeable par InsuranceContract.escalade_threshold_pct)Historique — par certificat
subscription_limit_exceedednouveau La soumission ferait dépasser le plafond NN300 cumulé (used_limit + valeur > subscription_limit)Renforcement — cumulé
certificates_limit_reachednouveau Le contrat a atteint son nombre maximal de certificats émisRenforcement — cumulé
Changement de comportement : avant renforcement, le dépassement des plafonds cumulés (subscription_limit, certificates_limit) provoquait un blocage HTTP dur à la création du certificat (InsuranceContract::canIssue()). Depuis le renforcement, seul le statut du contrat (actif/expiré) bloque la création (isActiveAndValid()) — le dépassement des plafonds cumulés déclenche désormais une escalade hiérarchique à la soumission, comme le mécanisme du « plein » déjà existant. canIssue() est conservé tel quel pour l'affichage du badge de statut sur le tableau de bord des plafonds.

Gestion des seuils & validations hiérarchiques par filiale

Nouvelle UI d'administration /admin/approvals/configs (ApprovalWorkflowController::configs/storeConfig/updateConfig/toggleConfig/destroyConfig) permettant à un admin_filiale (sa filiale) ou super_admin (toutes) de créer/modifier/activer/désactiver des règles : type de déclencheur, seuil (%), et jusqu'à 2 étapes de validation (rôle approbateur + délai en heures ouvrables).

// app/Models/ApprovalWorkflowConfig.php — evaluateCondition()
if (! empty($condition['subscription_limit_exceeded'])
    && ! $contract->exceedsSubscriptionLimit((float) $certificate->insured_value)) {
    return false;
}
if (! empty($condition['certificates_limit_reached'])
    && ! $contract->reachedCertificatesLimit()) {
    return false;
}

Cycle de vie d'une escalade

1. Déclenchement

CertificateController::submit() appelle ApprovalWorkflowService::triggerIfNeeded() — crée un ApprovalRequest (step 1) et notifie les approbateurs de l'étape.

2. Décision ou timeout

L'approbateur valide/rejette depuis /admin/approvals/{id}. Si le délai (due_date, en heures ouvrables) expire, nsia:check-escalades (horaire) fait automatiquement remonter à l'étape suivante, ou rejette si c'était la dernière étape.

3. Garde-fou anti-contournement

CertificateController::issue() refuse toute émission directe tant qu'un ApprovalRequest PENDING existe pour le certificat.

4. Émission automatique

À l'approbation de la dernière étape, le certificat passe directement à ISSUED et son PDF est généré (issueCertificate()).

Bug planification corrigé : la commande de vérification des délais était enregistrée sous app:check-escalades alors que routes/console.php planifiait nsia:check-escalades — mismatch silencieux qui empêchait toute auto-escalade/rejet sur timeout de s'exécuter. Toutes les commandes Artisan personnalisées de ce projet doivent utiliser le préfixe nsia: (vérifié systématiquement contre routes/console.php).

Impression sur souches physiques pré-imprimées

Système d'« overlay » : chaque champ variable du certificat est positionné en absolu (mm) par-dessus une page vierge, destinée à être imprimée sur le papier à souche déjà pré-imprimé par la filiale (numérotation, cadres, mentions légales). Sélection automatique du gabarit par code filiale (getTemplateForTenantCode()), mode calibration (?calibrate=1) affichant une grille de repère en mm pour ajuster les coordonnées contre le papier réel.

Coordonnées à valider physiquement : les positions livrées pour les 6 gabarits pays sont des valeurs de départ raisonnables, non calibrées contre un exemplaire physique réel (aucun accès imprimante/papier en environnement de développement). Un ajustement fin via le mode calibration est nécessaire avant mise en production.

DéploiementChecklist de mise en production

1. Préparer l'environnement

PHP 8.2+, PostgreSQL 16, Node 20+, Composer, queue worker (Supervisor)

2. Variables d'environnement

Copier .env.example → .env. Configurer DB, Mail, Microsoft/Google OAuth, Mindee, APP_KEY.

3. Dépendances

composer install --no-dev --optimize-autoloader
npm ci && npm run build

4. Base de données

php artisan migrate --force
Puis les seeders (voir §Scheduler) dans l'ordre : rôles → filiales → référentiel → workflows d'approbation → modèles de certificats.

5. Chiffrement PII (1 fois)

php artisan nsia:encrypt-sensitive-fields — à exécuter une seule fois après migration initiale des données.

6. Optimisations Laravel

php artisan config:cache && php artisan route:cache && php artisan view:cache

7. Stockage

php artisan storage:link — lien symbolique public/storage → storage/app/public (documents demandes partenaires, GUCE, logos)

8. Scheduler (crontab)

* * * * * php /path/to/artisan schedule:run >> /dev/null 2>&1

9. Queue worker (Supervisor)

Configurer Supervisor pour php artisan queue:work --sleep=3 --tries=3

10. Vérification finale

Tester login SSO, MFA, émission certificat, PDF, export, impression souche (mode calibration), extraction GUCE. Décider explicitement du sort de la RLS (rattacher tenant.context ou retirer la migration).

À trancher avant mise en production : le middleware SetTenantContext (RLS) n'est actuellement rattaché à aucune route. Si les politiques RLS PostgreSQL sont actives en base sans que app.current_tenant_id soit jamais défini, cela peut soit bloquer silencieusement des requêtes légitimes, soit ne rien appliquer du tout selon la politique par défaut (USING vs PERMISSIVE) — à valider avec un test d'intégration dédié avant go-live.
